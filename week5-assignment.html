Week 5 Assignment: Security, Availability & Reliability

⏱ ~90 minutes ✅ Free Tier 📚 Terraform + AWS Console

Overview

In this assignment, you will update your existing Terraform code from Week 4 to add security hardening, high availability through auto scaling, and reliability via automated backups. These enhancements directly apply the Cloud Security, High Availability, and Reliability concepts covered in the Week 5 lecture.

Objectives

  • Harden your infrastructure with proper Security Groups and Network ACLs
  • Configure EC2 Auto Scaling Groups for high availability
  • Set up an AWS Backup plan for EBS volume reliability
  • Verify your changes work end-to-end and clean up all resources

Prerequisites

  • Completed Week 4 Assignment (working Terraform code with VPC, EC2, EBS, S3)
  • AWS CLI configured with your credentials
  • Terraform installed and initialized
  • Your public IP address (find it at whatismyipaddress.com)

Step-by-Step Instructions

Part 1: Improved Security

1
Update your Security Groups and Network ACLs.
Modify your existing Terraform code so that your subnets are properly protected:
  • Your Security Groups should only allow SSH (port 22) access from your specific IP address — not from 0.0.0.0/0. Use a CIDR block like YOUR_IP/32.
  • Configure Network ACLs on your subnets to restrict traffic appropriately. The private subnet should not allow direct inbound traffic from the internet.
  • Ensure all other unnecessary ports are closed. Only open what is strictly needed.
💡 Tip: You can find your public IP by visiting whatismyipaddress.com. Use this IP in your Security Group ingress rule: cidr_blocks = ["YOUR_IP/32"]
2
Test your SSH connection.
After applying your Terraform changes, SSH into your EC2 instance to verify the Security Group is correctly configured. Take a screenshot of a successful SSH connection to include in your submission.
📸 SSH Screenshot

Your screenshot should show:

  • A successful SSH session connected to your EC2 instance
  • The instance hostname or IP visible in the terminal prompt

Part 2: Improved Availability

3
Configure Auto Scaling Groups for your EC2 instance.
Add the following resources to your Terraform code:
  • Create a Launch Template (or Launch Configuration) based on your existing EC2 instance configuration (AMI, instance type, security groups, key pair).
  • Create an Auto Scaling Group with:
    • Minimum instances: 1
    • Maximum instances: 4
    • Desired capacity: 1
  • Add a Target Tracking Scaling Policy that scales based on CPU utilization above 50% using ASGAverageCPUUtilization as the predefined metric.
📚 Reference: See the Terraform docs for aws_autoscaling_group and aws_autoscaling_policy.

Part 3: Improved Reliability

4
Set up an AWS Backup plan for your EBS volume.
Add the following Terraform resources:
  • An AWS Backup Vault to store your backups.
  • An AWS Backup Plan with a backup rule (e.g., daily backups with a retention period).
  • An AWS Backup Selection that targets your EBS volume.
  • An IAM Role for AWS Backup with the AWSBackupServiceRolePolicyForBackup managed policy attached.
⚠ Important: You can test deploying the backup plan to verify it works, but make sure you destroy the backup resources after testing to avoid any charges. AWS Backup is not covered under the free tier.

Submission

5
Upload your updated Terraform code to your homework directory as "Week 5 Assignment".
Include all .tf files. Please remove any access credentials (AWS access keys, secret keys) before uploading.
6
Upload a screenshot of your successful SSH connection (from Step 2).
🚨 Security Reminder: Never commit AWS credentials (access keys, secret keys) to any repository or shared directory. Use environment variables or the AWS CLI credential file instead.

Cleanup

7
Destroy all your created Terraform resources.
Run terraform destroy in your project directory and confirm the destruction when prompted. This will remove all the resources you created and ensure you don't incur any ongoing charges.
🗑 Cleanup Checklist
  1. Run terraform destroy and type yes to confirm
  2. Verify in the AWS Console that all resources have been removed (EC2, ASG, EBS, S3, Backup Vault)
  3. Check your AWS Billing dashboard to confirm no unexpected charges
