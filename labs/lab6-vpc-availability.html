Lab 6: Exploring VPC Infrastructure and Availability Zones

⏱ ~20 minutes ✅ Free Tier 📚 AWS Console Only

Overview

Amazon VPC is the networking foundation of your AWS environment. Every AWS account comes with a default VPC that spans multiple Availability Zones — this is the basis for building reliable, fault-tolerant architectures. In this lab, you'll explore VPC components, understand how subnets distribute across AZs, and create a security group. This connects to the Reliability pillar principles of "Networking, quotas, and constraints" and the AWS Global Infrastructure concepts (regions, AZs) from the Week 2 lecture.

Objectives

  • Understand VPC components: subnets, route tables, internet gateways, and security groups
  • See how the default VPC distributes subnets across multiple Availability Zones
  • Create a custom security group with specific inbound rules
  • Explore AWS service quotas to understand account limits

Prerequisites

  • An AWS account (free tier is sufficient)
  • Signed in to the AWS Management Console

Step-by-Step Instructions

1
Navigate to the VPC Dashboard.
In the AWS Console, use the search bar at the top and type VPC. Click on the VPC service to open it. Review the summary showing counts of VPCs, subnets, route tables, internet gateways, and other components.
2
Examine the default VPC.
Click on "VPCs" (or "Your VPCs") in the left sidebar, then click on the default VPC. Note the IPv4 CIDR block (typically 172.31.0.0/16). This is the IP address range for your entire VPC.
3
Explore subnets across Availability Zones.
Click on "Subnets" in the left sidebar. Observe that you have multiple subnets, each in a different Availability Zone (check the "Availability Zone" column). This is the foundation of multi-AZ reliability — if one AZ goes down, resources in other AZ subnets remain available.
4
Examine route tables.
Click on "Route Tables" in the left sidebar, then click on the default route table. Examine the Routes tab. You should see:
  • A "local" route — for traffic within the VPC
  • A route pointing to an Internet Gateway — for internet access (typically 0.0.0.0/0 → igw-xxx)
5
View the Internet Gateway.
Click on "Internet Gateways" in the left sidebar. Observe the default internet gateway attached to your VPC. This enables communication between your VPC and the internet.
6
Create a custom security group.
Navigate to "Security Groups" in the left sidebar, then click "Create security group":
  • Security group name: web-server-sg
  • Description: Allows HTTP and HTTPS inbound traffic
  • VPC: select the default VPC
  • Add inbound rule: Type = HTTP, Source = 0.0.0.0/0 (Anywhere IPv4)
  • Add inbound rule: Type = HTTPS, Source = 0.0.0.0/0 (Anywhere IPv4)
  • Leave outbound rules as default (All traffic)
  • Click "Create security group"
7
Review security group rules.
Review your new security group's Inbound rules tab to confirm HTTP (port 80) and HTTPS (port 443) are listed.
8
Explore Service Quotas.
Navigate to Service Quotas (search Service Quotas in the console), then search for "VPC". Explore quotas like "VPCs per Region" and "Subnets per VPC". These limits are important to understand when planning architectures.
Note: The default VPC is designed to get you started quickly. In production, you would create custom VPCs with carefully planned CIDR blocks, private subnets, NAT gateways, and more restrictive security groups.

Completion Screenshot

📷 What to capture:

Take two screenshots:

  • The VPC Subnets page showing subnets in multiple Availability Zones (the AZ column must be visible)
  • The custom "web-server-sg" security group's Inbound rules tab showing HTTP and HTTPS rules

Important: Your screenshot must include your AWS account name/ID visible in the top right corner of the AWS Console to verify completion.

Cleanup

🗑 Cleanup Steps:
  1. Navigate to VPC → Security Groups
  2. Select the "web-server-sg" security group
  3. Click Actions → Delete security group → confirm
Important: Do NOT delete the default VPC, default security group, or default subnets — these are used by other AWS services.

Key Takeaways

  • VPCs provide network isolation — your resources live in your own virtual network
  • Subnets across multiple Availability Zones are the foundation of high availability and fault tolerance
  • Security groups act as virtual firewalls, controlling inbound and outbound traffic at the instance level
  • Service quotas define the boundaries of your architecture — always check limits when planning
