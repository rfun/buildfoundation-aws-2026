Lab 8: Creating and Querying a DynamoDB Table

⏱ ~20 minutes ✅ Free Tier 📚 AWS Console Only

Overview

Amazon DynamoDB is a fully managed NoSQL database that delivers single-digit millisecond performance at any scale. In this lab, you'll create a table with a partition key and sort key, add items, run queries and scans, and explore capacity modes. This connects to the Performance Efficiency pillar principle of Data Management — optimizing data storage, access patterns, and performance from the Week 2 lecture.

Objectives

  • Create a DynamoDB table with a composite key (partition key + sort key)
  • Add and manage items in the table
  • Understand the difference between Query (efficient, targeted) and Scan (full table)
  • Compare On-demand vs Provisioned capacity modes for performance planning

Prerequisites

  • An AWS account (free tier is sufficient)
  • Signed in to the AWS Management Console

Step-by-Step Instructions

1
Navigate to DynamoDB.
In the AWS Console, use the search bar at the top and type DynamoDB. Click on the DynamoDB service to open it.
2
Create a new table.
Click "Create table" to begin setting up your DynamoDB table.
3
Set the table name.
Enter CourseStudents as the Table name.
4
Set the partition key.
For Partition key, enter StudentId and select String as the type.
5
Set the sort key.
For Sort key, enter CourseName and select String as the type.
6
Configure capacity mode and create the table.
Under Table settings, select "Customize settings". For Capacity mode, choose "On-demand". Click "Create table" and wait for the table status to become Active.
7
Navigate to item creation.
Click on "CourseStudents" to open the table. Click "Explore table items", then click "Create item".
8
Create the first item.
Enter the following values:
  • StudentId = student-001
  • CourseName = AWS-Foundations
Click "Add new attribute" → String → Name = Name, Value = Alice.
Click "Add new attribute" → Number → Name = Grade, Value = 95.
Click "Create item".
9
Create additional items.
Repeat the item creation process to add the following items:
  • StudentId: student-002, CourseName: AWS-Foundations, Name: Bob, Grade: 88
  • StudentId: student-001, CourseName: Cloud-Security, Name: Alice, Grade: 92
  • StudentId: student-003, CourseName: AWS-Foundations, Name: Charlie, Grade: 97
10
Run a Query.
In the "Scan or query items" section, switch from "Scan" to "Query". Enter the Partition key value = student-001 and click "Run". Observe that it returns only Alice's records (2 items) — this is an efficient, targeted lookup.
11
Run a Scan.
Switch back to "Scan" and click "Run". Observe that it returns ALL items in the table — this reads every item and is less efficient.
12
Explore monitoring and settings.
Go back to the table overview. Explore the "Monitor" tab to see read/write metrics, and the "Additional settings" tab to confirm the capacity mode is set to On-demand.
Query vs Scan: A Query uses the partition key to go directly to the data — it's fast and efficient. A Scan reads every item in the entire table — it works but gets expensive and slow as the table grows. Always design your keys to support Query access patterns.
Capacity Modes: On-demand mode automatically handles scaling — you pay per request. Provisioned mode lets you set fixed read/write capacity units — it's cheaper for predictable workloads but requires you to plan capacity. For this lab, On-demand is simplest.

Cleanup

🗑 Cleanup Steps:
  1. Navigate to DynamoDB → Tables
  2. Select "CourseStudents"
  3. Click "Delete" → confirm by typing delete → Delete table

Key Takeaways

  • DynamoDB delivers consistent, single-digit millisecond performance — a key Performance Efficiency service
  • Choosing the right key design (partition key + sort key) determines your query efficiency
  • Query is always preferred over Scan — design your access patterns around your partition key
  • On-demand mode provides automatic scaling for unpredictable workloads, while Provisioned mode optimizes cost for stable workloads
