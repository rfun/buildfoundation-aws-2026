Lab 3: IAM Users, Groups, and Least Privilege Policies

⏱ ~20 minutes ✅ Free Tier 📚 AWS Console Only

Overview

AWS Identity and Access Management (IAM) is the foundation of security in AWS. In this lab, you'll create an IAM group with a read-only policy, add a user to it, and use the IAM Policy Simulator to verify that least privilege access works correctly. This connects to the Security pillar principle "Implement a strong identity foundation" from the Week 2 lecture.

Objectives

  • Create IAM groups and users to organize identities
  • Attach AWS managed policies to implement least privilege
  • Use the IAM Policy Simulator to test and verify access permissions
  • Understand the difference between allowed and denied actions under ReadOnlyAccess

Prerequisites

  • An AWS account (free tier is sufficient)
  • Signed in to the AWS Management Console
  • Signed in as a user with IAM administrative permissions

Step-by-Step Instructions

1
Navigate to IAM in the AWS Console.
In the AWS Console, use the search bar at the top and type IAM. Click on the IAM service to open it.
2
Create a new User Group.
Go to User Groups in the left sidebar. Click Create group. For the group name, enter ReadOnlyTeam.
3
Attach the ReadOnlyAccess policy.
In the "Attach permissions policies" section, search for and select ReadOnlyAccess. Click Create group.
4
Create a new IAM user.
Go to Users in the left sidebar. Click Create user. For the user name, enter lab-student.
5
Skip console access.
Do NOT check "Provide user access to the AWS Management Console" — we're just creating the identity for this lab exercise.
6
Add user to the group.
Click Next. Under Add user to group, select ReadOnlyTeam. Click Create user.
7
Open the IAM Policy Simulator.
Navigate to the IAM Policy Simulator by searching Policy Simulator in the IAM console, or use the left sidebar.
8
Select the lab-student user.
In the Policy Simulator, select Users → lab-student from the left panel.
9
Simulate a read action on S3.
Select service Amazon S3 → select action GetObject → click Run Simulation. Observe that the result is "allowed".
10
Simulate a destructive action on S3.
Clear the selection, then select action DeleteBucket → click Run Simulation. Observe that the result is "denied".
11
Simulate a read action on EC2.
Change service to Amazon EC2 → select DescribeInstances → click Run Simulation. Observe the result is "allowed" (read action).
12
Simulate a destructive action on EC2.
Select TerminateInstances → click Run Simulation. Observe the result is "denied" (write/destructive action).
Note: Notice the pattern: ReadOnlyAccess allows all Describe*, Get*, List* actions (read operations) but denies Put*, Create*, Delete*, Terminate* actions (write operations). This is least privilege in action — granting only the minimum permissions needed.

Completion Screenshot

Take a screenshot of the IAM Policy Simulator showing:

  • At least one "allowed" and one "denied" result for the lab-student user
  • The user name and policy visible in the left panel

Important: Your screenshot must include your AWS account name/ID visible in the top right corner of the AWS Console to verify completion.

Cleanup

🗑 Cleanup Steps:
  1. Go to IAM → Users → select lab-student → Delete → confirm by typing the username
  2. Go to IAM → User Groups → select ReadOnlyTeam → Delete → confirm

Key Takeaways

  • IAM groups let you manage permissions for multiple users efficiently rather than per-user
  • AWS managed policies like ReadOnlyAccess provide pre-built least privilege templates
  • The Policy Simulator is a powerful tool for testing permissions without making actual API calls
  • Least privilege means granting only the permissions needed — read-only users shouldn't be able to delete resources
